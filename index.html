핵심 요구사항 분석

  • OMS 엑셀 변환 — 지정 폴더 내 다수 OMS 엑셀 파일을 일괄 읽기하고, 헤더 분석으로 체인별 양식을 자동 식별하여 이카운트 ERP 양식으로 변환
  • 매핑 테이블 관리 — OMS 거래처명↔이카운트 거래처 코드, OMS 품목명↔이카운트 품목 코드 매핑 UI 제공. 미매핑 데이터 보류 및 경고, 오류 목록 엑셀 다운로드
  • 송장 자동화 — 이카운트 출고 완료 데이터를 수집하여 로젠택배 송장 발행 요청, 발행된 송장번호 저장 및 이카운트 역전송, 프린터 연동 출력
  • 박스 분할 알고리즘 — 품목별 1박스당 최대 포장 수량 기준으로 필요 박스 수 자동 계산, 동일 수령인 분할 송장 자동 생성
  • 비기능 — 8,000 거래처 + 2,500 품목 지연 없는 검색/매핑, 추후 택배사·OMS 양식 확장 용이한 구조

핵심 기술 선택 근거

  • Python FastAPI — pandas/openpyxl로 엑셀 파싱 최적, 빠른 API 개발, 소스코드 가독성 높아 내부 수정 용이
  • React + Ant Design — 데이터 그리드(Table) 컴포넌트 특화, 대량 데이터 표시에 최적화된 가상 스크롤 내장
  • MySQL 8.0 — 8,000 거래처 + 2,500 품목 고속 검색, 인덱스 최적화, 확장성 확보

유사 프로젝트 경험 기반 효율적 개발

15개 이상 외부 API 연동 경험(NICE KYC, Seyfert/PayGate 결제 등)과 80+ 엔티티 규모 대량 데이터 처리, 배치 자동화(크론잡) 경험을 바탕으로 효율적으로 진행합니다.

PRD (Product Requirements Document)

프로젝트 요구사항을 정의하고 기능 범위를 명확히 합니다.

1. 프로젝트 배경 및 목적

안경 체인 OMS 프로그램에서 하루 약 300건의 주문을 엑셀로 다운로드하여 이카운트 ERP에 수기로 업로드하고 있으며, 출고 처리 후 하루 500~600건의 로젠택배 송장 역시 수기 또는 엑셀로 처리하고 있습니다.

이러한 반복적인 수작업을 자동화하여 업무 시간 단축과 오류 방지를 달성하는 것이 핵심 목표입니다. 특히 거래처/품목 매핑 관리 기능과 출고 수량에 따른 택배 박스 분할 송장 발행 로직이 필수적으로 구현되어야 합니다.

운영 방식: 실시간 자동화가 아닌, 사용자가 하루 2회 정도 수동으로 실행하는 배치(Batch) 형태

3. 기능 요구사항

3.1 OMS 주문 엑셀 변환 및 이카운트 연동

  • [필수] 지정 폴더 내 다수의 OMS 엑셀 파일 일괄 읽기 및 병합
  • [필수] 업로드된 엑셀의 헤더를 분석하여 체인별 양식 자동 식별 및 표준화
  • [필수] 변환된 데이터를 이카운트 주문 업로드용 엑셀 양식으로 다운로드
  • [선택] 이카운트 API로 직접 전송 (API 테스트 베드 환경에서 사전 검증)

3.2 매핑 테이블 및 오류 관리

  • [필수] OMS 거래처명 ↔ 이카운트 거래처 코드 매핑 UI
  • [필수] OMS 품목명 ↔ 이카운트 품목 코드 매핑 UI
  • [필수] 매핑 테이블에 없는 데이터 발생 시 변환 보류 및 경고 알림
  • [필수] 매핑 실패 건 엑셀 다운로드
  • [필수] 오류 목록에서 코드 지정 시 매핑 테이블 자동 업데이트

3.3 이카운트 출고 기반 로젠택배 송장 자동화

  • [필수] 이카운트 API로 특정 일자/조건의 출고 완료 데이터 자동 수집
  • [필수] 수집된 출고 데이터로 로젠택배 송장 발행 요청
  • [필수] 발행된 송장번호 저장 및 이카운트 ERP 역전송
  • [필수] 로젠택배 전용 프린터 연동 출력
  • [필수] 일자별 송장 발행 내역, 성공/실패 여부 조회 및 재발행

3.4 포장 박스 분할 알고리즘

  • [필수] 품목 마스터에 품목별 1박스당 최대 포장 수량 기준값 설정
  • [필수] 출고 수량을 포장 단위로 나누어 필요 박스 수 자동 계산
  • [필수] 산출된 박스 수량만큼 동일 수령인 정보로 분할 송장 자동 생성
  • [필수] 동일 수령인 다품목 출고 시 수량 합산하여 박스 수 계산
